# ECI 故障响应模板

## 模板 A：创建失败 / Pending 过久

1. 现象：`<资源ID>` 在 `<时间范围>` 持续 Pending 或创建失败。
2. 证据：
- `kubectl describe pod` / 控制台事件
- `DescribeContainerGroupStatus` / `DescribeContainerGroupEvents`
- 地域、可用区、规格、配额信息
3. 判断路径：
- 先配额与规格可用性
- 再网络（vSwitch/安全组/公网出口）
- 再镜像与存储
4. 修复建议：
- 调整规格或可用区
- 修正网络配置或镜像访问路径
- 补齐存储参数/权限
5. 依据链接：从 `official_link_baseline.md` 选择 2~4 条强相关链接。

## 模板 B：镜像拉取失败

1. 现象：`ImagePullBackOff` / `ErrImagePull`
2. 证据：
- Pod 事件关键行
- 镜像地址、仓库类型、鉴权方式
- 网络出口（EIP/NAT）与安全组
3. 判断路径：
- 镜像地址是否可达
- 凭证是否有效
- 是否需要 ImageCache
4. 修复建议：按最小改动优先，给出回滚路径。
5. 依据链接：镜像、网络、日志相关官方页。

## 模板 C：存储挂载失败

1. 现象：`MountFailed` 或容器启动后读写异常
2. 证据：
- 事件与容器日志
- 卷类型（NAS/云盘/OSS）与挂载参数
- 网络连通与权限信息
3. 判断路径：
- 卷类型与场景匹配
- 参数和权限正确性
- 网络依赖是否满足
4. 修复建议：给出具体参数改动点与验证步骤。
5. 依据链接：`overview-of-volumes` + 对应卷类型页面。

## 模板 D：日志/监控缺失

1. 现象：日志为空或监控指标缺失/延迟
2. 证据：
- 日志采集配置（环境变量/SLS CRD）
- 监控接入配置与时间窗口
- 相关 API 返回
3. 判断路径：
- 先采集开关与配置
- 再权限与网络
- 最后核对指标口径与统计周期
4. 修复建议：先恢复可观测性，再处理业务故障。
5. 依据链接：日志采集与监控指标官方页。
